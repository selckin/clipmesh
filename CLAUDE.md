# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

clipmesh is a Rust daemon that syncs Wayland clipboards across a LAN mesh of hosts over Noise-encrypted TCP. Copy on one host, paste on all. See `README.md` for user-facing behavior and configuration.

## Commands

```bash
./build.sh                  # every CI gate, in CI order -- run before pushing
cargo build                 # debug build
cargo build --release       # release build (CI gate)
cargo test                  # all tests (lib unit tests + tests/two_nodes.rs + tests/paste.rs)
cargo test --lib fswatch    # one module's tests
cargo test --lib config::tests::load_reports_a_broken_config_symlink   # one test
cargo test --test two_nodes # one integration-test file (likewise --test paste)
cargo clippy --all-targets -- -D warnings   # lint (CI gate; warnings are errors)
cargo fmt --check           # format check (CI gate)
cargo run -- --config ./examples/config.toml   # run the daemon
```

MSRV is Rust 1.80 (`Cargo.toml` `rust-version`). The binary is normally the long-running daemon, but a few flags are one-shot and exit: `--allow <glob>`, `--deny <glob>`, `--rules` (edit/inspect the MIME-rules file), `--sync-config` (rewrite the config as the canonical commented template — see `config_template.rs`), plus `--config <path>`. `--paste` (or invoking the binary via a `wl-paste` symlink) is a separate **wl-paste impersonation** mode that pulls a node's clipboard over the mesh and prints it (see `paste.rs`).

Running the **real** clipboard backend needs a live Wayland compositor implementing `ext-data-control-v1`/`zwlr-data-control-v1`. All automated tests use the mock clipboard instead, so they run headless in CI; the Wayland path is verified manually.

## Architecture

A node is assembled in `node::spawn_node` (called from `main`): it binds a listener + accept loop (bounded by a `Semaphore`), spawns one `dial_loop` per configured peer, builds the shared `MimeRules`, and starts the `SyncEngine`. Peers form a **full mesh** — every node dials every other; clipmesh never forwards between peers, so each node must list all others directly.

The stack, bottom to top:

- **`transport.rs`** — Noise `NNpsk0` (`Noise_NNpsk0_25519_ChaChaPoly_BLAKE2s`) keyed by the preshared key. `handshake` splits a stream into `SendHalf`/`RecvHalf` sharing a *stateless* transport (independent per-direction nonces) so the two halves run in separate tasks. Records are chunked to Noise's 64 KiB cap, and each goes out as **one** `write_all` (length prefix encrypted into the same scratch buffer) with a single flush per logical message — `set_nodelay` is on, so a separate 4-byte header write became its own TCP segment per record, ~512 of them for a 32 MiB message. Reassembly grows the receive buffer to twice the bytes **actually delivered**, floored at a step and capped at the message: never to the length the peer merely *claimed*, which with 64 inbound slots would let a few hundred bytes of traffic pin gigabytes.
- **`protocol.rs`** — wire messages `Message::{Hello, Clip, Rules, Get, GetReply}`, encoded with **bincode**. `Hello` carries a `PeerRole` (`Peer` or `Paster`) so a one-shot `--paste` client is never mistaken for a mesh member; `Get`/`GetReply` are the paste request/response (see `paste.rs`). `Offer = IndexMap<String, Vec<u8>>` (one clipboard state's MIME representations, kept in the source compositor's advertise/preference order end-to-end); `content_hash` (BLAKE3) sorts a copy internally so identity/dedup stays order-independent even when the compositor hands types back reordered. **`Hashed` pairs an `Offer` with its `content_hash` and is what the engine passes around** — `Hashed::new` is `content_hash`'s only production caller. The offer is immutable behind it, so a stale hash is unrepresentable: a transform that changes the content builds a new `Hashed`, one that doesn't returns its input and carries the hash forward. That is what lets the inbound path reuse the hash it already verified against the wire, and it replaces the "did this stage change anything?" flag nobody would have kept correct. The offer sits behind an `Arc`, and **terminal consumers take `into_arc`, not `into_offer`** — `Message::Clip.offer` and `Clipboard::write_offer` both carry `Arc<Offer>`. That is the whole point of the sharing: the engine's batch holds every planned action's content alive at once, so an owned terminal parameter finds refcount > 1 and deep-copies, *moving* the copy rather than removing it. serde's `rc` feature serializes the pointee, so this is byte-identical on the wire (pinned by a round-trip test). `into_offer` remains for the transform steps that genuinely take the map apart. `type_matches`, `TEXT_PLAIN`, `is_text` and `is_text_plain` live here too, because each is a contract *between* the two ends (the node narrows a `-t` read by `type_matches`; the client then picks out of the reply by the same rule).
- **`peer.rs`** — `run_connection`: Noise handshake + `Hello` exchange (checks `PROTOCOL_VERSION` and rejects a self-connection via the `SelfConnection` marker error, which dial loops treat as permanent), then reader/writer tasks. `AbortGuard` tears the connection's child tasks down on cancel.
- **`mesh.rs`** — `Mesh`: live peer table keyed by remote node ID. Duplicate connections to one peer are expected (both nodes dial each other); index 0 of each peer's group is the designated sender, the rest are receive-only warm standbys. First connection to a peer fires `connect_tx`, which the engine turns into a resync. A `PeerRole::Paster` lives in a **separate `clients` table**, not in `peers`: `broadcast` and `peer_count` read `peers`, so a one-shot paste client is invisible to them by construction rather than by a filter each new reader has to remember, and it can never land at index 0 of a real peer's group (which would have silently swallowed every send to that peer). `send_frame_to` is the only reader consulting both, so a paster is still reachable for its reply.
- **`sync.rs`** — `SyncEngine`, the brain.
  - **Engine loop:** `run` selects over the local clipboard watch, inbound mesh messages, peer-connect events, and rules-changed pings. Handles echo suppression (compare incoming vs. current hash per selection), debounce, `Direction` (send/receive/both), payload caps, and content filtering. **Ordering everywhere — live updates and reconnect resync alike — is the hybrid logical clock `Version { stamp, origin }`** (`protocol.rs`, held by `ContentState`); higher stamp wins, `origin` (node ID) breaks ties. `Version` travels the wire *as one field* on both `Clip` and `Rules`, deliberately: with the pair loose, a site could compare `stamp` alone, which compiles fine and silently diverges the mesh on a tie. Compare whole `Version`s — construct one only where a stamp is genuinely minted (`tick()`). Password-manager contents (`x-kde-passwordManagerHint = secret`) are dropped before broadcast (and, while `exclude_sensitive` is on, never re-owned). **`Pipeline::{Broadcast, Inbound, Own, Mirror, Serve}` is the single taxonomy of "why is this content moving"**, and `Pipeline::stages()` the one place it turns into transforms; `Provenance` is the deliberately narrower write-only subset (`plan_batch` must not be able to emit "broadcast" as a write reason) and converts into it. Previously the two write pipelines mapped through `Provenance::stages` while the rest were `Stages` constants hand-picked at call sites, so each new consumer went around the mapping — `Serve` was added that way. `apply_stages` returns `Result<Hashed, Unavailable>`: the reason nothing survived is a value, because a `--paste` client is told it over the wire and "the clipboard is empty" is a bad answer to "every type you offered is denied by my rules".
  - **Capture-side features (both optional):** `synthesize_text_plain` back-fills `text/plain` (+`;charset=utf-8`) from a legacy `UTF8_STRING`/`STRING`/`TEXT` atom when none exists, and `take_ownership` re-offers each watched selection so clipmesh owns it (surviving the source app) — and, with synthesis, makes the back-filled `text/plain` paste on the origin host too.
  - **Local selection bridge (`link_selections`):** optionally mirrors the local CLIPBOARD and SELECTION on one host — distinct from `sync_selection` (which syncs each selection across the mesh). Local propagation runs as a deterministic read → plan → execute batch pipeline (`handle_batch`): read each changed selection once into `read` (the batch's whole view of the clipboard, echoes included), a pure `plan_batch` decides every broadcast and every write (each selection written at most once, with `Own`/`Mirror` provenance) up front, then `execute_write` performs them. **The plan names its content instead of carrying it** — an `Action` is `{ target, source }`, where `source` is the changed selection supplying the payload (itself, or the bridge source for a mirrored partner). So `plan_batch` takes only the list of changed *kinds* and copies no clipboard payload to reach a decision; execute looks each one up in `read`. The bridge mirrors only *local* user changes: it reconciles against the partner's actual content (so an out-of-band drift is re-mirrored), and a concurrent direct change to the partner in the same batch is never clobbered (direct-change-wins, expressed structurally in `plan_batch`).
  - **Echo suppression of the engine's own writes:** content the engine itself wrote — an inbound mesh apply, content restored at startup, an ownership rewrite, or a bridge mirror — is recorded by `ClipboardIo` and its watch echo is dropped one-shot during `handle_batch`'s classification, so engine-written content is never re-broadcast, re-bridged, or re-owned (this is what makes propagation terminate). **`clipboard::io::ClipboardIo` owns the backend privately, in a module the engine cannot see into, so `Clipboard::write_offer` is literally unreachable from `SyncEngine`** — the only way to write is `io.write`, which records the marker before writing and rolls it back on failure. A new engine-write path therefore *cannot* forget the record; this was previously a convention enforced only by a doc comment on `write_selection`, and the failure mode is an echo storm. Same treatment `Hashed` gives stale hashes. **The read half is structural too:** the marker must be consumed by exactly one reader (the batch classifier) and by no other, so that choice is a *method* — `io.read_classified` reads and consumes, returning an `Origin::{User, Echo}`; `io.read` never consumes. There is no free-standing "take the marker", so a new reader cannot land on the wrong side of a rule whose only symptom is a mesh that never settles. A mesh-synced mirror target is broadcast **directly** by `plan_batch` (not via its echo), so a synced partner is still fed to the mesh like any other local change. See `docs/superpowers/specs/2026-06-20-bridge-write-consolidation-design.md`.
- **`clipboard/`** — the `Clipboard` trait (`watch`/`list_types`/`read_offer`/`write_offer`). `list_types` and `read_offer`'s `only` argument exist because a content read is *not* cheap here — the Wayland backend spends a connection and a roundtrip **per representation** — so a caller that wants one type, or only the names, must be able to say so before the reads happen. `wayland.rs` is the real in-process backend; `watch.rs` is its change listener (runs on a dedicated thread because Wayland dispatch blocks); `mock.rs` backs every test (`block_reads`/`allow_reads` gate reads, which is what makes the `Initial`-contract tests exact rather than hopeful). `io.rs` is `ClipboardIo`, the engine's sole gateway (see the echo-suppression bullet). `atoms.rs` owns X11/ICCCM selection-atom semantics — which atoms carry content, which are pure machinery, and what encoding each declares — because that is neither an engine policy nor a Wayland detail; `wayland.rs` and `sync.rs` both consume it instead of each keeping its own overlapping-but-different list.
- **`paste.rs`** — the `wl-paste` impersonation mode (`--paste`, or a `wl-paste`-named symlink).
  - **Fetch mechanism:** `fetch_offer` dials a node via `peer::run_connection` (a throwaway `Mesh` + a random node ID) announcing itself as `PeerRole::Paster`, then **asks**: `Message::Get { kind, want }`, answered with `GetReply`. `fetch_from_any` races `fetch_offer` across every target via a `tokio::task::JoinSet` and returns the first that responds; `resolve_targets` makes the default (no `--node`) every configured peer, `--node` a single one.
  - **Why a request, not the old push-scrape:** it used to take whatever `Clip` the node happened to push on connect. That cost the serving node a full resync per `wl-paste` (a rules snapshot plus a live capture of *every* synced selection — on Wayland one connection per advertised MIME type), transferred the whole offer even for `-t text/plain`, and could only ever report a timeout listing four unrelated causes, because the client had never asked anything. `GetWant::{All, One, TypesOnly}` (from `narrowing`) lets the node send only what the invocation needs, and `GetResult::{Offer, Types, NotOffered, Unavailable}` lets every failure name itself (`interpret`/`explain`). `Unavailable::{NotSynced, Empty, Sensitive, Denied, TooLarge, Unreadable}` are the outcomes the serving node's pipeline already distinguishes — it logs a different line for each — so returning the reason as a value is what lets the asking side say something true instead of "empty" for all six. `resync_on_connect` is no longer a paste prerequisite; only `p.send` (direction + `sync_selection`) gates what a node will serve.
  - **The narrowing reaches the backend, not just the wire.** `serve_get` maps `GetWant` onto `Clipboard::list_types` / `read_offer(only)`, so `-l` reads no representation contents at all and `-t` reads exactly one. Narrowing only the reply would leave `wl-paste -l` pulling a 30 MB image off the compositor — one connection per type — to print a list of names. `listing_types_reads_no_representation_contents` pins this by gating content reads in the mock and requiring the answer anyway.
  - **But every request starts from the advertised *names*, because narrowing the read destroys two decisions.** A narrowed read filters the offer down before the pipeline sees it — which strips the `x-kde-passwordManagerHint` marker (`-t text/plain` served a password a broadcast refuses, and `-l` skipped the content pipeline entirely) and strips the legacy X11 atom that `synthesize_text_plain` derives from (the same clipboard answered `text/plain` over the mesh and "not offered" to `-t text/plain`). Both decisions are answerable from names, so `serve_get` lists types first — one names-only roundtrip, no contents — and then: `serving_is_sensitive` gates *every* shape of request before any content is read (reading only the tiny hint representation, and only when it is advertised, so ordinary content pays nothing); `serve_source` picks the atom to read when the requested `text/plain` would be synthesized rather than offered; `narrow_to` trims the reply back to the request afterwards. The type list is then reused by the failure path instead of being fetched again.
  - **`NotOffered` is emitted only when the requested type really is absent** from what this node would serve — `narrowed_failure` checks that rather than assuming it. Collapsing every narrowed failure into `NotOffered` produced replies that listed the requested type as available while claiming it was not offered, and discarded the `Unavailable` reason the un-narrowed path reports correctly.
  - **A query must not record.** Serving uses `Stages::SERVE`, not `BROADCAST`: same content filters (a pull must not expose what a push wouldn't), but `RulesStage::Apply` instead of `Record`. `Record` appends unseen types, persists the rules file and broadcasts it mesh-wide — so serving through it would let any paster make this node rewrite its rules and push them to every peer.
  - **Output & wiring:** pure helpers (`select_type`/`list_types`/`render`/`output_bytes`) decide the output; `run` wires `Config::load` and the wl-paste flags, writing raw bytes to stdout. `main` detects the mode (argv[0] or `--paste`) before the daemon flag loop.
- **`mime.rs`** — `MimeRules`: a **program-managed** TOML file (via `toml_edit`). Per-type allow/deny with case-insensitive globs (most-specific match wins, deny-by-default). clipmesh auto-appends unknown types, sorts on save, and stamps a `[clipmesh]` version table so the file can be shared mesh-wide under whole-file last-writer-wins (`share_mime_rules`). The built-in `TEMPLATE` carries the default rules — the common text/image allows plus the deny globs (`JAVA_DATATRANSFER*`, `x-special/*`, …) that stop per-transfer atom names accumulating forever — and **`examples/mimetypes` is generated from it** (`render_example`, pinned by a golden test, same arrangement as `examples/config.toml`). Never hand-edit the example; change `TEMPLATE`, then `CLIPMESH_REGEN_EXAMPLE=1 cargo test --lib example_mimetypes_matches_template`. `reload_if_changed` compares **content, not mtime**, so clipmesh's own writes don't trigger a reload→rebroadcast loop. **Two constructors, because healing is destructive:** `MimeRules::open` never writes and returns a `RulesFileState::{Missing, Empty, Malformed, Unreadable}` when it can't load, while `load` is `open` plus "rewrite everything except a file we merely failed to read". The CLI (`--rules`/`--allow`/`--deny`) uses `open` so it cannot clobber a file it couldn't parse — that guard used to live in `main.rs` as a second read and a second `toml_edit` parse, outside the abstraction, which left both a TOCTOU window and a classification that could disagree with `read_file`'s. All writes funnel through one private `persist_body`, which uses `fsutil::write_atomic` (temp file + rename onto the resolved symlink target): this file is rewritten on every unseen MIME type and is shared mesh-wide, so a truncated half-file from a crash or full disk would be adopted by peers.
- **`config_template.rs`** — the canonical config-file template + the `--sync-config` normalizer: one ordered list of `Block`s describes every option, its comment, and its default; `render` emits the file overlaying the user's present values (options the user set stay active, the rest become commented defaults). `examples/config.toml` is **generated** from the same template and pinned by a golden test — never hand-edit the example (the test fails the build); change the template, then `CLIPMESH_REGEN_EXAMPLE=1 cargo test --lib example_config_matches_template`. Adding a config option means adding a `Block` here too: `render` only emits keys that have one, so a missing block would make `--sync-config` silently drop that option from a user's file. `every_config_option_has_a_template_block` enforces this against `config::RAW_CONFIG_KEYS`, and `sync_config` refuses to write a lossy file rather than trusting the test alone. Each optional block declares its shown value as `Shown::Default` — a *promise* that uncommenting the line changes nothing, enforced by `commented_defaults_are_the_parsers_defaults`, which activates the whole template and compares the parsed `Config` against a minimal one — or `Shown::Sample` (illustrative; `peers` and `mime_rules_file` have no literal default to print, and their comments say so). That split exists because both had silently drifted from the real defaults with the golden test still green. `key_defaults` is the single exhaustive scan of `TEMPLATE`, with no `_` arm, so a new `Block` variant stops the build rather than falling out of a derived set. Two mechanisms keep that key list honest, because either alone has a hole: `Config::from_toml` destructures `RawConfig` **without a `..` rest pattern**, so a new field is a compile error, and `raw_config_keys_matches_the_struct` pins `RAW_CONFIG_KEYS` to serde's own field list (captured from `deserialize_struct`, not scraped from an error message) so satisfying the compiler by editing only the destructure still fails.
- **`config.rs` / `fswatch.rs`** — `Config::load` parses TOML and defaults `mime_rules_path` to sit beside the config. `fswatch` is an inotify watcher (dedicated thread) over both files, and is symlink-aware (follows a symlinked config/rules file to its real target). A **config** change makes the process exit cleanly so a supervisor restarts it (most settings can't hot-apply — this requires the systemd unit or any restart-on-exit supervisor); a **rules** change reloads in place. **`fswatch` only reports, though — it never exits.** Both arms now have the same shape: the rules change reloads and pings, the config change is sent to `main`, which owns `ConfigChange`/`config_change_action` and decides to restart. A watcher thread is the wrong place to end the process, and deciding it there also forced this module to parse the config (`use crate::config::Config` is gone). The watches sit on the *directories*, so a re-stow (`rm -rf` + recreate) makes the kernel drop them and report `IN_IGNORED` — which carries no file name and so must be handled before the name guard, or the entries keep a dead descriptor and `read_events_blocking` blocks forever on an empty watch list instead of erroring, leaving the process permanently deaf with nothing logged and no restart. It re-registers in place, or returns the error so `supervise` retries on its backoff until the directory is back.
- **`backoff.rs`** — the shared exponential-backoff helper (`next_delay`), the watcher restart policy (`RESTART_*` / `watcher_restart`), and `supervise`, the restart *loop* both long-lived watcher threads run on: it owns the timing, the "was that run healthy?" reset and the restart log, so `fswatch` and `clipboard::watch` supply only the attempt (returning `ControlFlow::Break` to retire the watcher, as the clipboard watcher does when the engine is gone). The dial loop in `node.rs` intentionally keeps its own schedule: a much lower cap (5s, so a returning LAN peer reconnects fast), a longer healthy threshold, and jitter to desynchronise simultaneous mesh reconnects.

### Cross-cutting invariants

- **bincode is not self-describing.** Any change to `Message` or its fields must bump `protocol::PROTOCOL_VERSION`; mismatched nodes are refused at handshake rather than failing to decode later. A bump means every node in the mesh must be upgraded — an old node and a new one simply refuse each other, so a partial rollout stops syncing rather than corrupting anything.
- **tokio async** carries networking and the engine; **blocking work runs on dedicated `std::thread`s** (inotify in `fswatch`, Wayland dispatch in `clipboard::watch`) and reports back over mpsc channels. Rules-file I/O goes through `SyncEngine::with_rules`, which runs the closure on the blocking pool — the engine is one task driving `run`'s select loop, so `std::fs` there stalls inbound messages and peer connects too. The one deliberate exception is the startup rules-version read, which runs before the loop, so there is nothing to stall.
- **Startup content comes from the backend, never from a read.** `Clipboard::watch` must deliver one `ClipboardEvent::Initial` per non-empty selection, carrying its content as of the subscribe, before any `Changed` for it. The engine reads *lazily*, after an event, so anything it reads reflects "now" — it cannot tell restored content from a copy made a moment after startup, and guessing wrong suppresses the copy and records it at stamp 0 where a peer's older clipboard outranks it. Only the backend exists at subscribe time, so only the backend can answer. `Initial` is adopted by `adopt_restored` (stamp 0, never broadcast/bridged/re-owned); `Changed` is a user action. `MockClipboard` honours this exactly by snapshotting and registering under both locks in `local_copy`'s order, which is what makes `a_copy_racing_startup_is_not_mistaken_for_restored_content` deterministic. The Wayland watcher reports `Initial` from its own startup roundtrip; the residual gap there is marked KNOWN GAP in `clipboard/watch.rs` and closes when content reads move onto the watcher's own connection.
- Shared state crosses the async/thread boundary as `Arc<Mutex<...>>` (e.g. `MimeRules` is shared between `SyncEngine` and `fswatch`).
